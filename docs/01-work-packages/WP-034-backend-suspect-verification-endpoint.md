# backend suspect verification endpoint

## Objective

Implement the first backend-owned suspect verification endpoint for Sequel City Web Detective.

The endpoint must allow the web runtime to verify a submitted suspect name through the existing database-backed `Solution` trigger without exposing general learner `INSERT` access through the SQL editor.

## Scope

### In Scope

- Add backend types for suspect verification request and response payloads.
- Add a backend service that submits a suspect name to the existing `Solution` table and reads the deterministic verdict generated by the database trigger.
- Add a Fastify route for `POST /api/case/verify-suspect`.
- Register the route in the existing backend app.
- Add focused backend service and route tests.
- Update the API package test script to run the new tests.
- Update implemented API documentation to include the new case verification endpoint.
- Update current architecture, release-readiness, user-journey, and testing docs where they currently say suspect verification is not implemented.

### Out of Scope

- Frontend suspect verification UI.
- Full deterministic case progression service.
- Notebook persistence.
- Authentication or authorization.
- SQL editor mutation support.
- Database schema or trigger changes.
- Hard-coding hidden suspect names in application code.
- Runtime AI behavior.
- Production or cloud deployment behavior.

## Constraints

- Preserve backend authority for verification.
- Preserve frontend presentation-only boundaries.
- Preserve learner SQL editor read-only behavior.
- Use the database-backed `Solution` table and trigger as the verification authority.
- Do not hard-code murderer or mastermind names in application code.
- Keep route handlers thin and delegate verification logic to a backend service.
- Use parameterized SQL for suspect submission.
- Do not introduce new dependencies.
- Do not introduce runtime AI behavior.
- Document implemented behavior only.

## Required Behavior

### API Contract

Add:

```http
POST /api/case/verify-suspect
```

Request body:

```json
{
  "suspect": "Full Name"
}
```

Successful verification response:

```json
{
  "success": true,
  "data": {
    "suspect": "Full Name",
    "verdict": "Database-generated verdict text"
  },
  "message": "Suspect verification completed."
}
```

Malformed request response:

```json
{
  "success": false,
  "message": "Request body must include a non-empty string `suspect` field."
}
```

Verification failure response:

```json
{
  "success": false,
  "message": "Suspect verification failed. Verify the suspect name and database connection."
}
```

### Backend Service

- Trim the submitted suspect name before verification.
- Reject empty suspect names before touching the database.
- Insert the suspect into `Solution` using a parameterized query.
- Read the latest non-null verdict for that suspect from `Solution`.
- Return the suspect and verdict without interpreting hidden correctness in application code.
- Return deterministic failure messaging when database verification fails or no verdict is found.

### Route Behavior

- `POST /api/case/verify-suspect` must return `400` for missing, non-string, or empty `suspect`.
- The route must delegate successful request handling to the verification service.
- Service failures must return `500` with the deterministic failure response.
- The route must not expose stack traces or raw database errors.

### Documentation Behavior

- API docs must list the new implemented endpoint.
- Architecture docs must state that suspect verification is now implemented as a backend-owned database-backed route.
- Release-readiness docs must no longer list suspect verification endpoints as absent.
- User-journey docs must describe suspect verification as backend-owned but must not add frontend UI claims that are not implemented.
- Testing docs must remove suspect verification endpoint testing from excluded scope and describe current backend/API verification expectations.
