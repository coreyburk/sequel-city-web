# Session Data Collection Guide

## Purpose

This guide defines how to collect user testing evidence for Sequel City Web sessions so later observation reports and cross-session analysis are based on real session data rather than inference or fabrication.

## When to Use This Guide

Use this guide when preparing for, running, observing, or cleaning up any moderated user testing session that will later produce observation notes, issue lists, observation reports, or cross-session analysis.

Use this guide before Session-002 and Session-003 evidence capture begins.

## Required Materials

- Session script or task list
- Evidence capture template for the correct session
- Note-taking document for facilitator and observer
- Screen recording tool, if approved for the session
- Audio recording tool, if approved for the session
- Timekeeping method
- Test environment access
- Stable internet connection
- Working application build for the session
- Contact method for backup support if the session environment fails

## Facilitator Responsibilities

- Prepare the session environment before the tester joins.
- Confirm the correct task sequence will be used.
- Explain the session format and task flow.
- Prompt the tester to think aloud when appropriate.
- Avoid coaching unless help is required to keep the session moving.
- Record whether facilitator help was required for each task.
- Mark task outcome for each task as `Yes`, `No`, or `With Help`.
- Keep note capture focused on observed behavior.
- Preserve raw evidence for later analysis.

## Observer Responsibilities

- Capture detailed notes during the session.
- Record only what was seen or heard.
- Separate direct observation from any later interpretation.
- Capture direct quotes when possible.
- Capture hesitation, confusion, backtracking, or repeated attempts.
- Record when facilitator help was provided and why.
- Record task outcome without inferring causes.
- Flag issue candidates as candidates only, not confirmed findings.

## Evidence Collection Rules

- Record only observed behavior.
- Separate observation from interpretation.
- Capture direct quotes when possible.
- Capture hesitation or confusion points.
- Capture whether facilitator help was required.
- Capture task outcome.
- Do not invent findings.
- Do not generalize from a single tester.
- Do not propose solutions during note capture.
- Do not convert issue candidates into conclusions during the session.
- Do not fill blank evidence fields with assumptions after the fact.

## Session Setup Checklist

- Confirm the correct session evidence template is ready.
- Confirm the six required tasks are listed in the template.
- Confirm date, start time, and participant metadata fields are available.
- Confirm the test environment is available and stable.
- Confirm required recording permissions and tools, if used.
- Confirm facilitator and observer know their roles.
- Confirm note-taking conventions and file naming conventions.
- Confirm all prior session files are closed to avoid mixing evidence.
- Confirm the tester starts from the intended initial state.

## During-Session Note-Taking Guidance

- Write notes in chronological order.
- Use short, factual statements.
- Prefer concrete actions over summaries.
- Record visible actions such as clicks, pauses, rereads, navigation changes, and query edits.
- Capture the approximate time for each task.
- Record the exact point where hesitation or confusion appears.
- Mark when the tester asks for help or when the facilitator intervenes.
- Capture direct quotes in quotation marks when possible.
- Keep observation notes separate from issue candidates.
- If interpretation seems useful, place it outside the raw evidence record for later analysis, not in live note capture.

## After-Session Cleanup Checklist

- Confirm all session metadata fields are complete.
- Confirm each required task has an outcome recorded.
- Confirm approximate time is captured for each task, if available.
- Confirm observation notes are readable and specific.
- Confirm direct quotes are preserved as written or marked approximate if reconstructed immediately after the session.
- Confirm facilitator help events are documented.
- Confirm issue candidates remain phrased as candidates, not findings.
- Confirm unanswered fields are intentionally left blank rather than guessed.
- Save the evidence file using the required naming convention.
- Store any approved recordings or supporting artifacts with the session evidence.

## Required Evidence Before Analysis

Before observation reports or cross-session analysis are created, the session package must include:

- Completed session metadata
- Environment details
- Facilitator identification
- Tester role
- Task evidence for all six required tasks
- Observation notes
- Direct quotes, if available
- Issue candidates
- Post-test responses
- Overall session outcome
- Evidence completeness checklist

If the required evidence does not exist, analysis must be deferred rather than fabricated.

## File Naming Conventions

- Session guide: `Session-Data-Collection-Guide.md`
- Session evidence capture files: `Session-00X-Evidence-Capture.md`
- Future observation reports should keep the session number in the filename.
- Future cross-session analysis files should not be created until all required evidence exists.

## Quality Standards for Observation Notes

- Notes must be factual, specific, and traceable to observed behavior.
- Notes must distinguish what happened from what it might mean.
- Notes should be detailed enough that another reviewer can understand the event without attending the session.
- Notes should include task context.
- Notes should preserve uncertainty instead of replacing it with confidence.
- Notes should identify whether the tester completed the task independently, failed, or needed help.

## Common Evidence Collection Mistakes to Avoid

- Writing conclusions instead of observations
- Rewording a tester quote into cleaner language that changes meaning
- Skipping hesitation or recovery steps because the task was eventually completed
- Omitting facilitator intervention details
- Marking issue candidates as confirmed usability problems
- Generalizing from one tester to all users
- Proposing design or product solutions during evidence capture
- Filling in missing evidence after the session from memory alone
- Creating analysis artifacts before the required evidence exists
