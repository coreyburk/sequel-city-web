# Case Verification Endpoints

Case verification is backend-owned and database-backed. It provides a controlled replacement for the original assignment's learner-written `INSERT INTO Solution` pattern without allowing general mutation through the query execution endpoint.

## `POST /api/case/verify-suspect`

Submits a suspect name for deterministic verification through the existing `Solution` table and database trigger.

### Request Body

| Field | Type | Required | Notes |
|---|---|---|---|
| `suspect` | `string` | Yes | Full suspect name submitted by the learner or frontend |

### Response Codes

| Status | When returned |
|---|---|
| `200` | Suspect verification completed and a database-generated verdict was returned |
| `400` | Request body does not include a non-empty string `suspect` field |
| `500` | Verification could not complete because the backend could not retrieve a database verdict |

### Success Shape

| Field | Type | Notes |
|---|---|---|
| `success` | `true` | Indicates the verification flow completed |
| `data.suspect` | `string` | Suspect name returned from the database-backed verification row |
| `data.verdict` | `string` | Verdict text generated by the database trigger |
| `message` | `string` | Current success message is `Suspect verification completed.` |

### Failure Shape

| Field | Type | Notes |
|---|---|---|
| `success` | `false` | Indicates verification did not complete |
| `message` | `string` | Deterministic validation or verification failure message |

The failure shape does not include a `data` object.

### Example Request

```json
{
  "suspect": "Full Name"
}
```

### Example `200` Success

```json
{
  "success": true,
  "data": {
    "suspect": "Full Name",
    "verdict": "Database-generated verdict text"
  },
  "message": "Suspect verification completed."
}
```

### Example `400` Malformed Request

```json
{
  "success": false,
  "message": "Request body must include a non-empty string `suspect` field."
}
```

### Example `500` Verification Failure

```json
{
  "success": false,
  "message": "Suspect verification failed. Verify the suspect name and database connection."
}
```

## Authority Notes

- The endpoint is the only implemented backend mutation path for suspect verification.
- The learner SQL editor remains read-only and still blocks general `INSERT` statements.
- The backend submits the suspect to SQL Server using parameterized SQL.
- SQL Server and the existing `Solution` trigger generate the verdict text.
- Application code does not hard-code hidden suspect identities.
- The current frontend provides a dedicated suspect verification panel and remains presentation-only.
